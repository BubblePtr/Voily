# 使用 PermissionFlow 引导 macOS 权限修复

## 背景

Voily 只需要两类系统权限：麦克风用于录音采集，辅助功能用于全局触发键监听和粘贴注入。麦克风的首次授权可以直接通过系统弹窗完成；辅助功能没有等价的系统弹窗，用户需要进入系统设置，并且在重新签名、更换安装位置或从不同构建安装后，授权记录也可能失效。

旧路径只在失败时给出提示，用户需要自己找到系统设置入口，尤其是辅助功能授权失败后容易误以为应用无法使用。

## 决策

- 引入 `PermissionFlow` 作为 App 层权限引导依赖，仅用于打开 macOS 隐私设置页并提示用户选择当前 App。
- 保留 `PermissionCoordinator` 负责麦克风授权请求和状态读取；不把业务权限判断下沉到第三方库。
- 新增 `AccessibilityPermissionGuide` 和 `MicrophonePermissionGuide`，把 PermissionFlow 调用集中在窄边界内。
- 设置页「输入」页提供完整权限检查卡片：首页只显示权限状态和缺失权限 banner，不直接做授权交互。
- 权限状态在相关页面 active 时轻量轮询刷新，用户从系统设置完成授权后不需要重新进入页面。
- Makefile 提供完整权限流程测试入口，测试前可重置本机相关权限。

## 放弃的方案

- **只保留错误提示**：实现最轻，但用户需要自己定位系统设置，辅助功能失败时困惑度高。
- **自己实现系统设置深链和说明 UI**：可控性高，但会重复维护 macOS 设置页路径、说明和选择 App 的细节。
- **把所有权限申请都交给 PermissionFlow**：边界过宽。麦克风首次授权仍应优先使用系统 API 弹窗，PermissionFlow 只作为设置页修复路径。

## 后果

正面：

- 迁移后，已有授权不需要重新申请；状态读取仍使用系统 API。
- 用户可以在设置页看到麦克风和辅助功能的当前状态，失败时有明确修复路径。
- 首页只提示状态和入口，不承载授权交互，避免打断数据概览。

负面：

- App 层新增 `PermissionFlow` 依赖，需要随上游兼容性变化维护。
- 权限状态刷新需要在相关页面 active 时做轻量轮询，不过只读取系统状态，不做重型工作。

迁移与回滚：

- 如果 PermissionFlow 出现兼容性问题，可以回滚到直接打开系统隐私设置页的实现，`SettingsPermissionActions` 和设置页 UI 不需要大改。
- 本地调试或验证完整权限流程时，先通过 `make prepare-debug` 重置麦克风和辅助功能授权，并注销、移除已安装副本，再安装并启动调试构建。

安全与隐私：

- PermissionFlow 不扩大 Voily 的权限范围，只引导用户打开系统设置。
- Voily 不读取系统权限列表，也不保存权限授权记录；状态只在运行时通过系统 API 查询。
- 权限缺失时，录音和文本注入路径会停在本地提示，不会绕过 macOS 隐私控制。
